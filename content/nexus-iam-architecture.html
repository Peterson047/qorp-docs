# Fluxo Completo de Permissões IAM Nexus Abaixo está a representação visual (Mermaid) e o passo a passo técnico da arquitetura que discutimos. ```mermaid graph TD subgraph "Camada de Entrada (Canal)" U[👤 Usuário: Maria] -->|Mensagem: 'Paga esse boleto'| API[🌐 API Nexus /chat] U -->|JWT Token (Google)| API end subgraph "Camada de Identidade (IAM)" API -->|Validar Token| IAM[🏛️ Identity Registry] IAM -->|Mapear Grupos Google| DB_USER[(📦 DB Users/Roles)] DB_USER -->|Cargo: Estagiário| IAM IAM -->|Filtra Deny Categories| POLICY[📜 Policy Engine] POLICY -->|Bloqueio: 'pagamentos', 'infra'| CONTEXT[🧠 Contexto de Permissão] end subgraph "Camada de Orquestração (Core)" CONTEXT -->|Contexto + Mensagem| ORQ[🏗️ Orquestrador] ORQ -->|Consulta Catálogo| CAT[📂 Tool Registry] CAT -->|Todas as Tools| FILTER{✂️ Filtro Dinâmico} subgraph "Filtragem de Ferramentas" FILTER -->|Tag: mercado| T1[✅ get_stock] FILTER -->|Tag: pagamentos| T2[❌ pay_invoice] FILTER -->|Tag: infra| T3[❌ drop_table] end T1 -->|Injeta apenas ferramentas permitidas| LLM[🤖 Modelo LLM] end subgraph "Camada de Execução (Skills)" LLM -->|Decisão: 'Não tenho ferramentas de pagamento'| RESP[💬 Resposta ao Usuário] RESP -->|Exibe no Chat| U T1 -.->|Se fosse Diretor| EXEC[⚙️ Execução MCP/Python] EXEC -.->|Log de Auditoria| AUDIT[(🛡️ Audit Trail)] end style T2 fill:#ffcccc,stroke:#ff0000 style T3 fill:#ffcccc,stroke:#ff0000 style T1 fill:#ccffcc,stroke:#00aa00 style IAM fill:#f9f,stroke:#333 style ORQ fill:#bbf,stroke:#333,stroke-width:4px ``` ## Passo a Passo Técnico (Simulação): ### 1. Entrada (OIDC) O usuário se autentica com sua conta corporativa do Google. O Nexus recebe um **JWT (JSON Web Token)** que contém o `email` e os `grupos` do Google Workspace. ### 2. O Mapeador (Identity Registry) O sistema consulta o seu banco (Postgres) e vê que o grupo `g-estagiarios-financeiro` do Google está mapeado para a Role `Estagiário` no Nexus. - **Contexto Gerado:** `user_id: maria_01`, `role: estagiario`, `denials: [pagamentos, infra]`. ### 3. O Filtro Dinâmico (Orquestrador) O Orquestrador (no seu código `nexus.py`) pega a lista global de 100 ferramentas e faz um filtro rápido: - *"Essa ferramenta tem a tag 'pagamentos'? Sim. A Maria tem essa tag bloqueada? Sim. Então, retiro da lista."* ### 4. Injeção no Modelo (Tool Binding) O LLM (Gemini) recebe a mensagem da Maria, mas na lista de ferramentas disponíveis para ele, **não existe** a função `pay_invoice`. - Para o modelo, é impossível sugerir o pagamento porque ele não "conhece" essa habilidade no contexto atual. ### 5. Auditoria (Audit Trail) Mesmo que o acesso seja negado, o Nexus grava no log: - `USER`: Maria | `ACTION`: Perguntou sobre pagamentos | `RESULT`: Negado por Policy (Tag: pagamentos). Isso garante que você possa auditar quem está tentando "forçar" a barra no sistema. 🏗️
