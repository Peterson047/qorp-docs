IAM & Controle de Acesso

Arquitetura de permissões hierárquicas, Policy Engine e filtro dinâmico de ferramentas para o Nexus Multi-Agent.

Visão Geral

O IAM do Nexus opera em três camadas independentes. A ideia central é: a IA só enxerga as ferramentas que o usuário tem permissão para usar. Se a ferramenta não está no contexto do agente, ele não pode tentá-la — nem por alucinação, nem por prompt injection.

Camada 01
Identidade
Quem é o usuário? Auth via Google OIDC. Mapeamento de e-mail para user_id interno.
Camada 02
Política
O que ele pode fazer? Policy Engine consulta a hierarquia e calcula as restrições acumuladas.
Camada 03
Execução
Orquestrador filtra as tools antes do bind_tools. LLM recebe apenas o subconjunto permitido.
ℹ
Filosofia: Deny List por cargo Funcionários têm acesso a tudo por padrão. Restrições são aplicadas por cargo ou individualmente. Toda ferramenta nova herda o comportamento da sua tag — sem precisar cadastrar regra por regra.

Hierarquia de Cargos

A empresa é modelada como uma árvore. Restrições aplicadas a um cargo são herdadas por todos os cargos abaixo. Um bloqueio no Coordenador afeta Analista, Assistente e Estagiário automaticamente.

O que acontece em cada etapa

1
Validação do Token
O cliente envia a mensagem com Bearer Token. O Gateway rejeita tokens inválidos antes de qualquer processamento.
Header de exemplo
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...
2
Identity Registry — Quem é esse usuário?
O JWT é decodificado e o e-mail é mapeado para um user_id interno único. O mesmo usuário pode entrar via WhatsApp, web ou n8n e será sempre o mesmo user_id.
Mapeamento
JWT sub: 109823456789 → email: maria.silva@empresa.com
→ user_id interno: usr_maria_001 | cargo: Estagiário
3
Policy Engine — O que ela pode fazer?
O motor sobe a árvore de cargos coletando todas as negações acumuladas. Consulta primeiro user_denials e depois role_denials recursivamente.
Árvore percorrida para Maria
Estagiário → Assistente → Analista → Coordenador → Diretor → CEO
deny_tags acumuladas: ["internal", "financial", "payments", "admin"]
4
SecurityContext — O "passaporte"
O resultado é serializado em JSON e cacheado no Redis por 1h. Esse objeto é tudo que o Orquestrador precisa — sem consultar o banco a cada mensagem.
Objeto gerado
{ user_id: "usr_maria_001", role: "Estagiário", deny_tags: ["internal", "financial", "payments", "admin"] }
5
Filtro Dinâmico — O Orquestrador como Gatekeeper
O Orquestrador filtra a lista global de tools. Ferramentas com tags bloqueadas são removidas. O LLM recebe apenas o subconjunto permitido via bind_tools — ele literalmente não sabe que as outras existem.
Resultado do filtro
Maria (Estagiária): 8 tools → [web_search, get_address_from_cep, knowledge_base_search]
Pedro (Diretor): 8 tools → [web_search, get_address_from_cep, knowledge_base_search, store_user_fact, get_stock_price, bigquery_query, pay_invoice]

Policy Engine — Como Funciona

O Policy Engine resolve três perguntas em sequência. A primeira que retornar um resultado encerra a verificação — não há avaliação das perguntas seguintes.

Pergunta 1: Este user_id tem uma negação individual para esta ferramenta?
Fonte: tabela user_denials
↓ Se sim: NEGADO     Se não: próxima pergunta
Pergunta 2: O cargo do usuário (ou algum ancestral) tem negação para a tag desta ferramenta?
Fonte: tabela role_denials com recursão na árvore
↓ Se sim: NEGADO     Se não: próxima pergunta
Pergunta 3: Nenhuma regra encontrada. Qual é a sensitivity da tag desta ferramenta?
Fonte: campo sensitivity em iam_tools
↓ public: PERMITIDO     restricted: NEGADO

Cenário 1 — Maria (Estagiária) tenta ver o faturamento

Bloqueado Ferramenta: get_revenue  ·  Tag: financial
Pergunta 1
Maria tem user_denial para get_revenue? → Não. Continua.
Pergunta 2
O cargo Estagiário tem role_denial para a tag financial? → Sim. Regra encontrada.
Resultado
NEGADO — ferramenta não injetada no contexto.
O que a IA diz
"Não tenho ferramentas para acessar dados de faturamento."

Cenário 2 — Pedro (Diretor) tenta pagar uma fatura

Permitido Ferramenta: pay_invoice  ·  Tag: payments
Pergunta 1
Pedro tem user_denial para pay_invoice? → Não. Continua.
Pergunta 2
O cargo Diretor tem role_denial para a tag payments? → Não. Continua.
Pergunta 3
Sensitivity da tag payments é restricted, mas Diretor (access_level 80) atende o nível mínimo. → Permitido.
Resultado
PERMITIDO — ferramenta injetada. Agente pode executá-la.

Cenário 3 — Ana (Analista) com exceção individual

Bloqueado por exceção Analista normalmente acessa internal — mas Ana tem um user_denial específico
Contexto
Ana é Analista (tem acesso a tools internal) mas foi bloqueada individualmente de store_user_fact por decisão do gestor.
Pergunta 1
Ana tem user_denial para store_user_fact? → Sim. Regra encontrada.
Resultado
NEGADO — exceção individual sobrescreve a permissão do cargo.

Tags de Ferramentas

Cada ferramenta é classificada com uma sensitivity tag. Novas ferramentas herdam o comportamento da tag automaticamente — sem precisar cadastrar regra por regra.

TagSensitivityDefaultNível mínimoExemplos
publicPúblicaallowEstagiárioweb_search, get_address_from_cep, knowledge_base_search
internalInternaallowAnalistastore_user_fact, search_user_memory, get_stock_price
financialFinanceirodenyCoordenadorget_revenue, get_balance, bigquery_query
paymentsPagamentosdenyDiretorpay_invoice, approve_payment, transfer_funds
adminAdministraçãodenyCEOdelete_database, manage_users, admin_panel
⚠
Ferramenta sem tag cadastrada Se uma ferramenta for criada sem tag no banco, o sistema usa public como padrão. Sempre cadastre a tag antes de publicar uma nova tool em produção.

03-iam-schema.sqlPostgreSQL
-- Cargos com auto-referência (árvore hierárquica)
CREATE TABLE IF NOT EXISTS roles (
  id              UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
  name            TEXT    UNIQUE NOT NULL,
  parent_role_id  UUID    REFERENCES roles(id),  -- quem está acima
  access_level    INTEGER NOT NULL,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Usuários mapeados a um cargo
CREATE TABLE IF NOT EXISTS iam_users (
  id          UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     TEXT    UNIQUE NOT NULL,
  email       TEXT    UNIQUE NOT NULL,
  google_sub  TEXT    UNIQUE,
  role_id     UUID    NOT NULL REFERENCES roles(id),
  is_active   BOOLEAN DEFAULT TRUE,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Ferramentas com sensitivity tag
CREATE TABLE IF NOT EXISTS iam_tools (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tool_name   TEXT UNIQUE NOT NULL,
  tag         TEXT NOT NULL,
  sensitivity TEXT NOT NULL DEFAULT 'public',  -- 'public' | 'restricted'
  description TEXT,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Negações por cargo (herdadas pela árvore)
CREATE TABLE IF NOT EXISTS role_denials (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  role_id       UUID NOT NULL REFERENCES roles(id),
  deny_tag      TEXT,
  deny_tool_id  UUID REFERENCES iam_tools(id),
  reason        TEXT,
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  CHECK (deny_tag IS NOT NULL OR deny_tool_id IS NOT NULL)
);

-- Exceções individuais de usuário
CREATE TABLE IF NOT EXISTS user_denials (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       TEXT NOT NULL REFERENCES iam_users(user_id),
  deny_tag      TEXT,
  deny_tool_id  UUID REFERENCES iam_tools(id),
  reason        TEXT,
  expires_at    TIMESTAMPTZ,  -- suporte a negações temporárias
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  CHECK (deny_tag IS NOT NULL OR deny_tool_id IS NOT NULL)
);

-- Log de decisões de permissão
CREATE TABLE IF NOT EXISTS iam_permission_logs (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     TEXT NOT NULL,
  tool_name   TEXT NOT NULL,
  decision    TEXT NOT NULL,   -- 'allowed' | 'denied'
  reason      TEXT,              -- 'user_denial' | 'role_denial' | 'default_deny'
  rule_source TEXT,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_iam_users_email  ON iam_users (email);
CREATE INDEX IF NOT EXISTS idx_role_denials      ON role_denials (role_id);
CREATE INDEX IF NOT EXISTS idx_user_denials      ON user_denials (user_id);
CREATE INDEX IF NOT EXISTS idx_perm_logs         ON iam_permission_logs (user_id, created_at DESC);

Código — Policy Engine & Filtro

src/iam/policy_engine.pyPython
from dataclasses import dataclass, field
from typing import List
import asyncpg, json

@dataclass
class SecurityContext:
    user_id: str
    role: str
    access_level: int
    deny_tags: List[str] = field(default_factory=list)
    deny_tools: List[str] = field(default_factory=list)

    def is_tool_allowed(self, tool_name: str, tool_tag: str) -> bool:
        # Prioridade 1: negação individual
        if tool_name in self.deny_tools:
            return False
        # Prioridade 2: negação por tag do cargo
        if tool_tag in self.deny_tags:
            return False
        return True


class PolicyEngine:
    def __init__(self, db_pool: asyncpg.Pool, redis):
        self.db    = db_pool
        self.redis = redis

    async def get_security_context(self, user_id: str) -> SecurityContext:
        cache_key = f"iam:ctx:{user_id}"
        cached = await self.redis.get(cache_key)
        if cached:
            return SecurityContext(**json.loads(cached))

        ctx = await self._build_from_db(user_id)
        await self.redis.setex(cache_key, 3600, json.dumps(ctx.__dict__))
        return ctx

    async def _build_from_db(self, user_id: str) -> SecurityContext:
        async with self.db.acquire() as conn:
            user = await conn.fetchrow("""
                SELECT u.user_id, r.name as role, r.access_level, r.id as role_id
                FROM iam_users u JOIN roles r ON r.id = u.role_id
                WHERE u.user_id = $1 AND u.is_active = true
            """, user_id)

            if not user:
                raise PermissionError(f"Usuário {user_id} não encontrado")

            # Percorre toda a árvore de cargos recursivamente
            role_denials = await conn.fetch("""
                WITH RECURSIVE role_tree AS (
                    SELECT id FROM roles WHERE id = $1
                    UNION ALL
                    SELECT r.id FROM roles r
                    JOIN role_tree rt
                      ON r.id = (SELECT parent_role_id FROM roles WHERE id = rt.id)
                )
                SELECT DISTINCT rd.deny_tag, t.tool_name as deny_tool
                FROM role_denials rd
                LEFT JOIN iam_tools t ON t.id = rd.deny_tool_id
                WHERE rd.role_id IN (SELECT id FROM role_tree)
            """, user['role_id'])

            user_denials = await conn.fetch("""
                SELECT ud.deny_tag, t.tool_name as deny_tool
                FROM user_denials ud
                LEFT JOIN iam_tools t ON t.id = ud.deny_tool_id
                WHERE ud.user_id = $1
                  AND (ud.expires_at IS NULL OR ud.expires_at > NOW())
            """, user_id)

            all_denials = [*role_denials, *user_denials]
            return SecurityContext(
                user_id=user['user_id'],
                role=user['role'],
                access_level=user['access_level'],
                deny_tags= list({r['deny_tag']  for r in all_denials if r['deny_tag']}),
                deny_tools=list({r['deny_tool'] for r in all_denials if r['deny_tool']}),
            )

# Mapeamento tool → tag (pode ser carregado do banco via iam_tools)
TOOL_TAG_MAP = {
    "web_search":            "public",
    "knowledge_base_search": "public",
    "get_address_from_cep":  "public",
    "store_user_fact":       "internal",
    "search_user_memory":    "internal",
    "get_stock_price":       "internal",
    "bigquery_query":        "financial",
    "pay_invoice":           "payments",
    "admin_panel":           "admin",
}

async def _call_agent(state, config, base_llm, all_tools, policy_engine):
    user_id = config["configurable"].get("user_id", "anonymous")

    # 1. Busca SecurityContext (Redis cache → Postgres)
    ctx = await policy_engine.get_security_context(user_id)

    # 2. Remove tools bloqueadas — o agente não sabe que existem
    filtered_tools = [
        t for t in all_tools
        if ctx.is_tool_allowed(t.name, TOOL_TAG_MAP.get(t.name, "public"))
    ]

    # 3. bind_tools dinâmico para essa chamada específica
    dynamic_llm = base_llm.bind_tools(filtered_tools)
    return await call_agent(state, config, dynamic_llm)

Riscos & Mitigações

⚠
Confused Deputy — o risco mais sério O agente tem permissão para usar a ferramenta, mas o usuário não deveria ver os dados que ela retorna. O Agente de Finanças consulta o BigQuery com a credencial do sistema — o escopo dos dados não é validado pelo IAM.

Solução: A ferramenta final deve receber o user_id e filtrar no nível da query: WHERE departamento = $user_scope.
⚡
Latência do bind dinâmico Reconstruir o bind_tools a cada mensagem pode impactar o TTFT com muitas ferramentas.

Solução: SecurityContext cacheado no Redis (TTL 1h). Quando um cargo muda, invalide com DEL iam:ctx:{user_id} para todos os usuários afetados.
🔍
Dificuldade de depuração "Por que a IA não usou aquela ferramenta?" sem rastreabilidade vira um pesadelo.

Solução: A tabela iam_permission_logs registra cada decisão com a regra que disparou. Endpoint de diagnóstico: GET /iam/debug/{user_id}/{tool_name}.
✓
Imune a prompt injection Ferramentas restritas não existem no contexto do LLM — não há validação em runtime para bypassar. Um usuário mal-intencionado não pode convencer a IA a usar uma ferramenta que ela literalmente desconhece.
